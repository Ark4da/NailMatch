# AGENTS.md

## Purpose

This file defines the project standard for `NailMatch`.
The repository is still at bootstrap stage, so the stack and workflow below are the chosen defaults that all future implementation should follow unless this file is updated intentionally.

Product direction:

- public website for manicure inspiration search;
- users upload manicure photos;
- the system stores images in a shared library;
- AI returns visually similar manicure options from the internal database;
- the browser stores a local profile of recent user uploads and generated concepts for style continuity;
- no auth, admin panel, marketplace, or social features in MVP.

## Project Stack

Use this stack as the project baseline:

- frontend: `Next.js 15` with App Router;
- language: `TypeScript` in strict mode;
- styling: `Tailwind CSS 4`;
- UI primitives: `shadcn/ui` on top of `Radix UI` only when needed;
- backend: `Next.js` route handlers for MVP;
- database: `PostgreSQL` via `Supabase`;
- file storage: `Supabase Storage`;
- vector search: `pgvector` in Supabase Postgres;
- AI provider: `OpenAI`;
- embedding model: `text-embedding-3-small` with 1536 dimensions;
- deployment platform: `Railway`;
- package manager: `pnpm`;
- forms: `react-hook-form` with `zod`;
- ORM / database access: `Drizzle ORM`;
- testing: `Vitest` for unit tests, `Playwright` for end-to-end tests;
- linting: `ESLint`;
- formatting: `Prettier`.

## Directory Conventions

Use this structure once the app is scaffolded:

- `src/app` for routes, layouts, pages, and route handlers;
- `src/components` for reusable UI components;
- `src/features` for product-specific modules such as upload, search, and gallery;
- `src/lib` for shared utilities, API clients, env parsing, and helpers;
- `src/db` for schema, migrations config, and query helpers;
- `src/types` for shared TypeScript types;
- `public` for static assets;
- `tests/e2e` for Playwright tests;
- `tests/unit` for Vitest tests.

Avoid dumping business logic into route files or UI components when it can live in `src/features` or `src/lib`.

## Key Commands

These commands are the required project interface. When the app is initialized, scripts must exist with these names:

- install dependencies: `pnpm install`
- run local development server: `pnpm dev`
- create production build: `pnpm build`
- run production server locally: `pnpm start`
- run linter: `pnpm lint`
- run formatter check: `pnpm format:check`
- apply formatting: `pnpm format`
- run unit tests: `pnpm test`
- run e2e tests: `pnpm test:e2e`
- run all required checks before merge: `pnpm check`

Deployment target:

- production hosting: `Railway`
- app must boot correctly from `pnpm start`
- production build must be generated by `pnpm build`

Expected `pnpm check` scope:

- type check;
- lint;
- unit tests;
- build.

## Code Rules

- Use `TypeScript` everywhere. Do not add plain JavaScript files unless there is a strong technical reason.
- Prefer server components by default in Next.js. Use client components only when browser interactivity is required.
- When writing complex features or significant refactors, use an ExecPlan (as described in PLANS.md) from design to implementation.
- Keep components small and focused. Move data logic, AI calls, and DB operations out of UI files.
- Validate all external input with `zod`, especially uploads, query params, and API payloads.
- Never trust client-provided metadata for uploaded images.
- Keep environment variable access centralized in one validated env module.
- Use absolute or alias imports consistently once configured. Do not mix many import styles.
- Do not hardcode secrets, URLs, bucket names, model names, or SQL connection details inside components.
- Prefer explicit return types for exported functions and shared utilities.
- Add comments only where logic is non-obvious. Avoid noisy comments.
- Keep files ASCII by default unless non-ASCII is required by product content.
- Do not introduce a second state-management solution unless the current approach is clearly insufficient.
- Do not add a separate backend service for MVP unless Next.js route handlers become a real bottleneck.
- Keep the app stateless so it can run safely in Railway containers.
- Do not rely on local disk for persistent user data; persistent assets must live in external storage.
- Treat browser localStorage profile data as convenience-only state, not as durable account data.

## Railway Rules

- The application must be deployable as a single `Next.js` service on `Railway`.
- Use environment variables for all secrets and runtime configuration.
- Do not assume a writable persistent filesystem in production.
- Do not depend on in-memory state for long-running jobs or durable processing.
- Prefer async processing patterns that can be retried safely if a Railway instance restarts.
- Keep startup simple: install dependencies, build the app, run `pnpm start`.
- If background work becomes necessary, define it explicitly as a separate Railway worker rather than hiding it inside web requests.

## AI and Search Rules

- Store the original uploaded image and a thumbnail.
- Generate and persist an embedding for every accepted manicure image.
- Search similar designs by vector similarity from the internal database only.
- Return a generated manicure concept image plus short human-readable context for the closest references.
- Treat external image-search references as optional inspiration only; do not copy, scrape, or persist external content as owned data.
- Prefer SerpAPI for Google Images references; use Bing Image Search as a fallback provider when configured.
- Keep AI prompts and model configuration centralized, versioned, and easy to update.
- Log failed AI processing without exposing secret values.
- Gracefully handle empty or low-volume databases by returning helpful fallback messaging.

## UX Rules

- MVP flow must stay simple: upload photo, wait for processing, view similar designs.
- Optimize for mobile first, then desktop.
- Show clear loading, success, empty, and failure states.
- Uploaded image preview must be visible before final submission when possible.
- The local profile should make past uploads and generated concepts visible without requiring auth.
- Search results should prioritize image clarity and fast scanning over dense metadata.
- Avoid login walls, complex onboarding, or dashboard-style interfaces in MVP.

## Self-Check Instructions

Before finishing any task, verify all items below:

- the change matches the chosen stack in this file;
- new scripts, tools, or infrastructure are reflected in `AGENTS.md`;
- lint passes;
- types pass;
- relevant tests pass or a clear reason is documented if tests could not run;
- build still succeeds for user-facing changes;
- the app remains compatible with Railway deployment assumptions;
- new environment variables are documented;
- no secrets were committed;
- no dead code, placeholder mocks, or debug logs remain unintentionally;
- error states and edge cases were considered, not just the happy path.

Minimum manual review before merge:

- check the main upload flow;
- check result rendering with at least one successful match;
- check empty-state behavior when the library has too few images;
- check mobile layout for upload and result screens;
- check that broken or invalid file uploads fail safely.
